# DOM I/O boundary audit

**Audit date:** 2026-07-15
**Frozen evidence snapshot:** `4df9a35e2bd1ac5b662382189e12973fc4e1c4c2` (`MB`)
**Scope:** all `144` MB `tests2/dom/**/*.test.ts` files, the v2-dom Vitest project, and `tests2/dom/_setup/{custom-elements,registry-bridge}.ts`.

## Rule and scope

The strict two-key rule is applied per assertion, not per file:

1. a qualifying MB E2E must cross the same real boundary and assert the material result; and
2. the unit assertion must remain meaningful with supplied I/O results.

Only MB `test:e2e:v2` Groups A/B/C and the twelve fixed Group-I integration files qualify. In particular, `tests/e2e/ui/search-e2e.spec.ts`, `preview-happy-path.spec.ts`, `dashboard-mutation-pending.spec.ts`, `goal-proposal-dismiss-reload.spec.ts`, `notification-policy.spec.ts`, `sidebar-keyboard-nav.spec.ts`, and `sidebar-navigation.spec.ts` were all `v2-browser`/non-daily at MB and are not evidence.

`happy-dom` is an in-process simulation. Its DOM, `Storage`, URL/history, custom-element registry, WebSocket substitutes, animation frames, and observer stubs are not browser/host I/O. Conversely, Node filesystem calls, module-file evaluation, real wall timers, and the opt-in performance clock remain boundaries. No DOM test at MB opens a real network socket or child process.

The baseline was checked with `git ls-tree`, `git show MB:<path> | nl -ba`, and the MB E2E runner/map. The denominator is `144` DOM files: `120` at `tests2/dom/`, `14` under `search/`, and `10` under `ui-fixtures/`.

## Transitive harness/setup behavior

At MB, `vitest.config.ts:231-259,335-343` runs v2-dom in `happy-dom` using the shared `pool:"forks"`, `isolate:false` project and includes every `tests2/dom/**/*.test.ts`. There is no v2-dom `setupFiles` entry.

At MB, `141` of `144` DOM files import `_setup/custom-elements.ts`; the standalone exceptions are `message-editor-arrows.test.ts`, `mobile-header-race.test.ts`, and `skills-page-scope.test.ts`. The setup file:

- pre-imports `safe-markdown-block` and `GateVerificationLive` (`:29-36`);
- records and replays custom-element definitions into each fresh happy-dom registry (`registry-bridge.ts:15-40,76-93`);
- pins lit-html's first `document` (`registry-bridge.ts:11,15-21`); and
- installs prototype `requestAnimationFrame`/`cancelAnimationFrame` fallbacks for teardown gaps (`registry-bridge.ts:42-74`). The fallback invokes stale callbacks only to clear shared module state and swallows rendering into torn-down DOM.

This is a simulated registry/window bridge plus a **real Node timer-leak interaction**. It performs no filesystem/network/process I/O itself, but its static imports are actual Vite/Node module evaluation from disk. A replacement must retain late-define mirroring, fresh-registry replay, lit's pinned-document ordering, `_renderScheduled` reset, and swallowed teardown-gap rendering. No qualifying E2E reproduces the cross-file `isolate:false` teardown gap, so this harness seam is **MB-PARTIAL**, never removable wholesale.

### Process, network, and host-I/O negative findings

- **Child/process:** no DOM test imports `node:child_process`, spawns a process/worker, or asserts exit/stdout/signals. `process.env` in search tests only selects delay/opt-in behavior; `process.on("unhandledRejection")` observes teardown failures.
- **Real network:** every fetch and WebSocket in DOM scope is stubbed/fake. There is no HTTP listener, socket, DNS, TLS, SSE, or WebSocket handshake.
- **Browser database:** no IndexedDB transaction runs.
- **Observers/workers:** no real observer delivery or worker execution runs.
