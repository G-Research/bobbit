## Findings from the real run (why A1/A2 failed â€” NOT the ledger)

Running **5 concurrent `test:v2` from a single worktree** corrupts state shared inside that one
`node_modules`, which has nothing to do with the ledger:

1. **`node_modules/.bin` wiped mid-run.** With `package-lock=false`, npm treats `node_modules`
   as out-of-sync and *reifies* (`.bin` removeâ†’recreate) on `npm run`; 5 concurrent reifies race
   and leave `.bin` empty â†’ reps 2 & 3 died instantly with `'vitest' is not recognized`.
   *Mitigated* by a canonical `npm ci` (now that the lock is correct).
2. **Vite dep-optimizer cache corruption.** All 5 vitest processes share
   `node_modules/.vite`; concurrent clear/rewrite produced spurious module-load failures
   (`TypeError: RemoteAgent is not a constructor` â€” a lazy import resolving to `undefined`),
   255 "failed" files in attempt 1.
3. **Isolating the Vite cache per run trades corruption for a CPU storm.** Giving each run its
   own cold `cacheDir` made all 5 optimize deps simultaneously â†’ each run ballooned to ~780 s
   (> the 600 s cap) and vitest still failed under the contention. A single `test:v2:core`
   with 2 forks + isolated cache passes cleanly (exit 0) â€” the failure is specifically the
   5-way same-worktree contention, not the tests.

**Conclusion:** "5 runs from one worktree" is not a realistic concurrency model. Real N-developer
concurrency = **N separate worktrees**, each with its own `node_modules` (warm Vite cache, no
`.bin` race) and harness-isolated ports â€” exactly what the cross-process ledger was built to
coordinate. The proof harness should spawn each concurrent `test:v2` from its own checkout.
