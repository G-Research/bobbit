transcripts to calibrate the synthetic fixture in

## 5. Recommendations for the harness fixture

Current `buildRealisticJsonl()` shape (medium = 50 msgs, see
`tests/manual-integration/perf-sidebar-nav.spec.ts`):

- 5–10 tool pairs spread evenly across 50 lines (so ~20–40% of messages are
  tool calls).
- One designated `>= 50 KB` blob per session.
- Tools sampled from `{bash, read, edit, grep, find, write, web_fetch}`
  in a deterministic round-robin (so each session uses ~7 distinct tools
  uniformly).
- Plain alternating `user / assistant` text on non-tool turns.
- No `thinking` blocks. No `compaction` lines. No `model_change` lines.

Concrete adjustments to match real corpora:

### 5.1 Tool-pair density

Real: **49% of message lines are `toolResult`**, ~36% of content blocks are
`toolCall`. That is **~1 tool pair every ~2 messages**, not 1 every ~5–10.

For `medium = 50 msgs`, target **18–22 tool pairs** (currently 5–10).
Equivalent rule: emit a tool pair on every other "step", interrupting only
for the occasional user/text turn.

### 5.2 Tool name mix (weighted, not round-robin)

Replace the uniform round-robin over a 7-tool array with weighted sampling
matching the corpus:

| Tool         | Weight (≈ share) |
|--------------|------------------|
| `bash`       | 35               |
| `read`       | 29               |
| `bash_bg`    | 12               |
| `edit`       | 9                |
| `grep`       | 7                |
| `write`      | 3                |
| `ls`         | 1                |
| `find`       | 1                |

(`bash_bg` is the most glaring omission today — it's the second-largest
heavy-tail blob producer.)

### 5.3 Role mix

Real: `user ≈ 6%, assistant ≈ 45%, toolResult ≈ 49%`. The fixture's strict
`user/assistant/user/assistant` alternation overstates user content by ~7×.
Most non-tool lines should be **assistant text**, with the rare user turn
roughly every ~17 messages.

### 5.4 `thinking` blocks

11.5% of all content blocks are `thinking`. The fixture emits zero. Add a
`thinking` block to roughly 1 in 3 assistant turns (matches sample) — even
a short thinking string flexes the renderer's collapsible-block path.

### 5.5 Tool-result size distribution

Current fixture: every non-blob result is a fixed-shape ~50–200 B string,
plus one explicit 60 KB blob per session.

Realistic distribution per tool result:

| Bucket             | Target frequency | Suggested body                       |
|--------------------|------------------|--------------------------------------|
| Tiny (≤ 500 B)     | ~30%             | One-line "ok"-style ACK              |
| Small (~1–2 KB)    | ~40%             | A few dozen short lines              |
| Medium (~5–10 KB)  | ~20%             | File-read-style body, 100–200 lines  |
| Large (~30–60 KB)  | ~7%              | Repeated multi-line stdout dump      |
| Heavy (~100–250 KB)| ~2%              | One per ~50 results — `read` / `bash_bg` shape |
| Outlier (~2 MB)    | ≤ 1 per session  | Mock `generate_image` base64 blob    |

The 1.6% > 50 KB frequency is the figure that matters for the lazy-load
feature flag — a `medium=50 msgs` fixture with ~20 tool pairs needs at
least 1 large + 1 outlier blob to exercise the threshold.

### 5.6 Header / lifecycle records

Each fixture file should include:

- One `type:"session"` header line at the top (cheap, but the reducer
  depends on it).
- A single `type:"model_change"` line near the top.
- For the `large` size variant, sprinkle 1–3 `type:"compaction"` lines
  (~25 KB each); the real corpus shows one compaction every ~30 messages
  in long sessions.

## 6. Anti-recommendations (don't model these)

- **Multi-block tool_result content.** Every >50 KB tool result in the
  corpus is a single text block. Don't synthesise multi-block tool results
  just for variety — it isn't realistic and complicates the renderer
  fixture without adding signal.
- **User-authored mega-messages.** Real users almost never paste >4 KB
  in a single user turn (user role only 5.7% of messages, all small).
  Keep `user` role lines under ~1 KB.
- **Heavy `image` blocks inline.** Only 11 `image` blocks in 9.5K
  messages (0.09%). The `generate_image` outliers come through as **text**
  blocks carrying base64, not as native `image` blocks. Don't bother
  modelling image-block-rendered avatars / inline images.
- **Long chains of `web_fetch` / `delegate` / `team_*`.** Sub-1% each,
  no heavy-tail. Don't waste fixture budget on them — keep them at most
  once per session for surface coverage.
- **Per-line variation in `model_change` / `thinking_level_change`.**
  These are tiny, fixed-shape control records. One of each per file is
  enough; don't randomise.
- **Synthetic `compaction` content beyond size.** Real compactions
  cluster around 13–28 KB and are otherwise opaque to the renderer; a
  fixed ~20 KB filler line is fine.
- **Adapting `small`/`medium`/`large` along the line-count axis only.**
  Real "heavy" sessions are heavy because of one or two **fat tool
  results**, not because of line count (note the 4.47 MB / 80-line and
  2.26 MB / 39-line outliers). The `large` variant should grow the blob
  budget, not just the message count.

## Methodology / footer

- Generator: `scripts/perf-profile-real-sessions.mjs` (new in this
  branch). Run with `node scripts/perf-profile-real-sessions.mjs
  <agent-sessions-root>`; defaults to `~/.bobbit/agent/sessions`. Output
  is JSON on stdout — pipe to a file and inspect with `jq` or feed back
  into this doc on re-baseline.
- Filter: directories whose name contains `bobbit-e2e`, `bobbit-manual-`,
  `bobbit-observe-`, or `bobbit-restart-` are excluded as test artefacts.
- Sample for line-level stats: **20 largest + 10 stratified mid-range
  files (>16 KB)**. p50/p95 figures in §1 are over all 1,940 files; p50/p95
  figures in §2–§4 are over the 30-file sample.
- Read-only run — nothing was written to `~/.bobbit/` or
  `C:/Users/jsubr/w/bobbit/.bobbit/`.
- No raw transcript content is included in this document. Large-blob
  "shape" classifications are heuristic (leading-character regex) plus
  originating tool name, not content text.
