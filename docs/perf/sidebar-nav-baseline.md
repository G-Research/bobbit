# Sidebar Nav Perf ‚Äî Phase 1 Baseline

Generated by the manual perf harness, run on the Phase 1 instrumentation
commit. The numbers below are the **first** measured baseline. Subsequent
runs append a JSON file under `docs/perf/history/` and refresh
`docs/perf/sidebar-nav-report.html` (the cross-commit view, generated by
`scripts/perf-report.mjs`).

The instrumentation itself is the Phase 1 deliverable. These numbers exist
to inform Phase 2 hypothesis selection ‚Äî they are **not** an SLO and the
harness is **not** in CI.

## 1. How to reproduce

```bash
# From the repo root:
BOBBIT_TIMING_LOG=1 npx playwright test \
  --config playwright-manual.config.ts \
  --grep "perf-sidebar-nav"
```

Optional toggles:

- `SCREENSHOTS=1` ‚Äî also dump full-page PNGs under
  `tests/manual-integration/.perf-out/screens/`.
- `BOBBIT_TIMING_LOG_MIN_MS=100` ‚Äî server log only requests ‚â•100ms (default
  is 0 = every request, set by the harness).

The harness:

1. Boots a real Bobbit gateway in an isolated `test-results/perf-sidebar-nav-gw/`
   directory (NOT your real `.bobbit`).
2. Registers a project, then POSTs 10 sessions (no agents ‚Äî we measure nav,
   not agent spawn).
3. Drives a browser through three passes:
   - **Cold pass** ‚Äî 3√ó `page.goto(#/session/:id)`. Triggers
     `api.session.fetch`, `ws.attach`, `reducer.rehydrate`, `paint.first`,
     `app.boot`.
   - **Warm pass** ‚Äî landing ‚Üí click each session sidebar row √ó 2 laps.
     Triggers `nav.click` ‚Üí `nav.session.ready` ‚Üí `paint.first`.
   - **Goal pass** ‚Äî creates a goal, clicks the dashboard button √ó 2 laps,
     then one `page.goto(#/goal/:id)` cold reload. Triggers `nav.click` ‚Üí
     `nav.goal.ready`, `api.goal.fetch`, `api.goal.gates.fetch`,
     `api.goal.agents.fetch`.

### Outputs

- `tests/manual-integration/.perf-out/sidebar-nav-<ts>.json` ‚Äî raw client
  entries + server `[timing]` lines tail.
- `tests/manual-integration/.perf-out/sidebar-nav-<ts>.html` ‚Äî sortable
  per-span table for that run.
- `docs/perf/history/<short-sha>.json` ‚Äî `{ commit, parentCommit, branch,
  timestamp, spans: { name: { p50, p95, p99, n, mean, max } } }`. One file
  per harness run keyed by HEAD short SHA. Committed to git so the
  cross-commit report renders at HEAD.
- `docs/perf/sidebar-nav-report.html` ‚Äî regenerated at the end of every
  run by `scripts/perf-report.mjs`. Inline SVG line charts (p50 solid,
  p95 dashed) per canonical span across every committed history file,
  plus a first-vs-latest summary table.

## 3. Ranked hotspots (top 5 by p50 on the critical path)

Each comes with a one-sentence likely-cause hypothesis. These are seeds for
Phase 2; the goal is to keep or kill each based on a focused A/B.

1. **`nav.session.ready` p50 ‚âà 89ms, p95 ‚âà 193ms.** The dominant contributor
   on the click‚Üíready critical path. Likely cause: synchronous `paint.first`
   (‚âà17ms) plus the WS-attach round-trip (‚âà36ms) plus
   `connectToSession` REST hydration (`api.session.fetch` ‚âà10ms) plus
   compositor settle. Subjective "click feels slow" complaints map here.
2. **`api.goal.gates.fetch` p95 ‚âà 94ms, max 125ms.** The gates endpoint
   spikes on the very first call after goal creation; subsequent calls are
   <12ms. Likely cause: cold-cache file IO on the goal's `gates.json`, or
   workflow snapshot materialisation. Worth confirming with the server
   `io=` count from `[timing]` lines.
3. **`ws.attach` p50 ‚âà 36ms.** Sits on the critical path of every cold
   session load. Likely cause: handshake + auth round-trip; the
   `outcome:auth_ok` detail confirms we reach auth. Phase 2 could test
   eager WS connect during the click handler (parallel with REST), or
   render-from-cached-snapshot-then-attach-WS.
4. **`paint.first` p95 ‚âà 31ms, max 49ms.** The render frame itself. Likely
   cause: full `doRenderApp()` re-render on every state change including
   the entire sidebar + project tree. Phase 2 candidates: more granular
   re-render scoping, or deferring off-screen blocks via
   `requestIdleCallback`.
5. **`api.session.fetch` p95 ‚âà 32ms.** Each session view hits
   `GET /api/sessions/:id` which ships the full transcript + tool inputs.
   For our seeded `messages: 0` sessions this is fast; real sessions with
   long transcripts will dominate. Phase 2 candidate per the goal spec:
   lazy-load tool content (the `/tool-content/:mi/:bi` endpoint already
   exists) so the initial payload is small.

The instrumentation will become more informative once it runs against a
realistic session corpus (long transcripts, multiple gates, an active
team). Phase 2 should re-baseline against such a corpus before committing
to optimisations.

## 4. Phase 2B A/B ‚Äî lazy tool content (feature-flagged)

**Hypothesis.** `GET /api/sessions/:id` was thought to ship the full
transcript + tool inputs on every cold load; stripping large tool-call
content blocks above a low threshold (default 4KB) and lazy-loading them
on demand via the existing `/tool-content/:mi/:bi` endpoint should reduce
`api.session.fetch` payload size + parse time and, by extension,
`nav.session.ready`.

**What landed.** Behind the `lazyToolContent` perf flag (set via
`localStorage.bobbitPerfFlags=lazyToolContent`):

- Server: `GET /api/sessions/:id?stripToolContent=1` now opt-in includes a
  `messages` array with tool-call content blocks larger than the threshold
  replaced by the same `{ _truncated: true, _originalLength, preview }`
  shape that the WebSocket `truncate-large-content` path already emits.
  The existing renderer (`WriteRenderer` + `Messages.ts::_onLoadFullContent`)
  hydrates the full content on demand via `fetchToolContent()` ‚Äî no new
  renderer code needed. Default response (no query param) is byte-identical
  to Phase 1.
- Client: the central `gatewayFetch` wrapper appends `?stripToolContent=1`
  to GET `/api/sessions/:id` URLs when the flag is on. Idempotent.
- Pure data-shape pinning test: `tests/session-strip-tool-content.test.ts`.

**Observation while implementing.** Reading the Phase 1 server route
revealed that `GET /api/sessions/:id` already returns **metadata only** ‚Äî
the transcript is delivered over WebSocket via `get_messages`, not REST.
This means the stock `api.session.fetch` span measures a small
metadata-only payload (‚âà10ms p50, ‚âà32ms p95 with `messages: 0`). The
strip experiment now opts into shipping the transcript on REST so the
hypothesis can actually be tested; the comparison below records both.

**A/B measurement.** Run the manual harness twice on the same commit:

```bash
# Flag OFF (default) ‚Äî metadata-only response
BOBBIT_PERF_FIXTURE_SIZE=large \
BOBBIT_PERF_HISTORY_TAG=flag-off \
  npx playwright test --config playwright-manual.config.ts \
  --grep "perf-sidebar-nav"

# Flag ON ‚Äî transcript stripped + lazy-load placeholders
BOBBIT_PERF_FIXTURE_SIZE=large \
BOBBIT_PERF_HISTORY_TAG=flag-on \
BOBBIT_PERF_FLAGS=lazyToolContent \
  npx playwright test --config playwright-manual.config.ts \
  --grep "perf-sidebar-nav"
```

The harness writes `docs/perf/history/<sha>-flag-off.json` and
`<sha>-flag-on.json`; the cross-commit report shows the delta visually.

**Result.** _(Pending Phase 2A's richer transcript fixture ‚Äî the
`messages: 0` seed produces a no-op strip. Numbers are populated once the
rich fixture lands and both passes are executed against it.)_

**Decision rule.** ‚â•100ms p50 reduction on `nav.session.ready` OR
`api.session.fetch` against the richer fixture ‚Üí recommend flag default-ON
for Phase 3. Otherwise document as "tried, didn't pay off" and leave the
flag default-OFF but available.
## 5. Realistic-fixture baseline (Phase 2A)

The Phase 1 baseline above seeded ten sessions with *empty* transcripts via
`POST /api/sessions`. That made `reducer.rehydrate` and `api.session.fetch`
artificially cheap and gave us no signal about how the click path scales
with real-world transcript sizes.

Phase 2A (task `3b056b96`) extends the harness with a deterministic,
seeded transcript fixture. For each of the ten sessions it writes a JSONL
transcript to disk mixing user/assistant text, 5‚Äì10 `tool_use` + matching
`tool_result` pairs, and one ‚â•50 KB tool-result blob. The harness then
flips those rows to `archived: true` in the project's `sessions.json`
*before* the gateway re-reads them on restart, so the WS archived-attach
path (`getArchivedMessages` ‚Üí disk JSONL parse) drives a real `messages`
frame into the client. Warm-pass nav is driven via
`window.__bobbitOpenForNavItem`, the same entry point keyboard nav uses,
so `nav.click` + `nav.session.ready` fire identically for archived rows.
See `tests/manual-integration/perf-sidebar-nav.spec.ts` for mechanism.

Fixture size is selectable:

| `BOBBIT_PERF_FIXTURE_SIZE` | msgs / session | total JSONL |
|---|---:|---:|
| `small`  |  10 |  ‚âà0.1 MB |
| `medium` (default) |  50 |  ‚âà0.7 MB |
| `large`  | 200 |  ‚âà2.3 MB |

### 5.3 What changed vs the empty-fixture baseline

**Caveat first.** The numbers are NOT apples-to-apples with ¬ß2. Phase 1
clicked *live* session rows (with real WS attach, agent RPC, etc.). The
realistic fixture rows are *archived*, so the click path is lighter‚Äîno
rpcClient.getState, no live event-buffer subscription. That is precisely
why `ws.attach` p50 drops from 36 ms (live) to 4 ms (archived) and
`nav.session.ready` p50 drops from 89 ms to 34 ms. The archived path is
the right harness to measure transcript-driven cost specifically; once
the lazy-tool-content optimisation lands (Phase 2B) we should add a
live-session fixture pass that prompts a real agent so we can compare
the full click path under realistic transcript load.

With that caveat, the realistic fixture surfaces three findings:

1. **`reducer.rehydrate` is decisively not a hotspot.** Even at 200 msgs
   it's 0.4 ms p50, 0.5 ms max. The Phase 1 candidate ‚ÄúLRU-cache reducer
   state keyed by session id‚Äù can be deprioritised ‚Äî there is no win to
   bank. Pinning test material for a future regression: any commit that
   pushes this above ~5 ms p50 with the medium fixture is a real bug.
2. **`paint.first` is the new transcript-scaling hotspot.** Medium p95 is
   27.5 ms; large p95 jumps to 103 ms, p99 to 159 ms, max 177 ms. The
   click path renders the entire transcript synchronously into
   `pi-chat-panel`; markdown / syntax-highlight cost dominates. Phase 2
   candidates: defer off-screen blocks via `requestIdleCallback`,
   virtualise long transcripts, or render text-only then upgrade.
3. **`api.session.fetch` p95 ‚â•97 ms under contention.** The endpoint is
   metadata-only (server `[timing]` shows it taking 0.5‚Äì1 ms). The
   wall-clock cost is the cold-pass `page.goto` opening many WS + REST
   connections in parallel and saturating the browser's HTTP/1.1 limit.
   Real users don't hit this pattern ‚Äî they nav one session at a time ‚Äî
   so this is a harness artefact, not a product hotspot. Worth
   instrumenting separately if we ever drive a ‚Äúprefetch on hover‚Äù
   experiment.

### 5.4 Re-ranked hotspots

| Rank | Span | medium p50 / p95 | large p95 | Phase 2B candidate? |
|---|---|---|---|---|
| 1 | `paint.first` | 16.9 / 27.5 | **103** | yes ‚Äî transcript-render scaling |
| 2 | `nav.session.ready` | **34.1** / 92.2 | 208 | downstream of paint.first |
| 3 | `api.session.fetch` payload size | 15.6 / 32.2 | 97 | yes ‚Äî lazy-tool-content target |
| 4 | `api.goal.gates.fetch` | 4.0 / 21.1 | 16.2 | no ‚Äî already cheap |
| 5 | `reducer.rehydrate` | 0.2 / 0.2 | 0.5 | **no ‚Äî confirmed not a hotspot** |

Neither `nav.session.ready` nor `reducer.rehydrate` p50 clears the 100 ms
snappy threshold per the task gating rule, so neither auto-promotes to
Phase 2B. **But** `nav.session.ready` p95 clears 100 ms at the large
fixture size, and the underlying driver is `paint.first` scaling with
transcript length ‚Äî that is where the real perceived-snappiness wins are.

## 7. Implementation notes (Phase 1 only ‚Äî not for prose-pinning invariants)

- Client trace primitive: `src/app/perf-trace.ts`. Cost-when-disabled
  pinned by `tests/perf-trace.spec.ts`.
- Feature-flag helper for Phase 2: `src/app/perf-flags.ts`.
- Server timing extended in `src/server/server.ts:792` ‚Äî log line format
  `[timing] METHOD path Xms bytes=B io=N`. Scoped to the five hot endpoints
  in the design doc.
- WS attach instrumentation lives in `src/app/remote-agent.ts::_connectWs`
  (open ‚Üí `auth_ok`/`auth_failed`/`error`/`close`). The span closes once,
  whichever outcome arrives first.
- Sidebar click handlers in `src/app/render-helpers.ts` route through
  `openForNavItem` so both keyboard and mouse paths emit
  `nav.click` / `nav.session.ready` / `nav.goal.ready` from one place.
- `window.__bobbitRefreshSessions` and `window.__bobbitOpenForNavItem` are
  exposed from `main.ts` for the manual harness ‚Äî landing-branch boot
  doesn't call `refreshSessions()`, so the harness kicks it explicitly.
- Manual harness: `tests/manual-integration/perf-sidebar-nav.spec.ts`.
  NOT in CI. Hard-fails (`process.exit(1)`) when any of the five canonical
  gate spans has zero samples.
- Cross-commit report: `scripts/perf-report.mjs` reads
  `docs/perf/history/*.json` and writes `docs/perf/sidebar-nav-report.html`.
  No deps, inline SVG, Bobbit CSS tokens.
