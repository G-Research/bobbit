# Sidebar Nav Perf — Phase 1 Baseline

Generated by the manual perf harness, run on the Phase 1 instrumentation
commit. The numbers below are the **first** measured baseline. Subsequent
runs append a JSON file under `docs/perf/history/` and refresh
`docs/perf/sidebar-nav-report.html` (the cross-commit view, generated by
`scripts/perf-report.mjs`).

The instrumentation itself is the Phase 1 deliverable. These numbers exist
to inform Phase 2 hypothesis selection — they are **not** an SLO and the
harness is **not** in CI.

## 1. How to reproduce

```bash
# From the repo root:
BOBBIT_TIMING_LOG=1 npx playwright test \
  --config playwright-manual.config.ts \
  --grep "perf-sidebar-nav"
```

Optional toggles:

- `SCREENSHOTS=1` — also dump full-page PNGs under
  `tests/manual-integration/.perf-out/screens/`.
- `BOBBIT_TIMING_LOG_MIN_MS=100` — server log only requests ≥100ms (default
  is 0 = every request, set by the harness).

The harness:

1. Boots a real Bobbit gateway in an isolated `test-results/perf-sidebar-nav-gw/`
   directory (NOT your real `.bobbit`).
2. Registers a project, then POSTs 10 sessions (no agents — we measure nav,
   not agent spawn).
3. Drives a browser through three passes:
   - **Cold pass** — 3× `page.goto(#/session/:id)`. Triggers
     `api.session.fetch`, `ws.attach`, `reducer.rehydrate`, `paint.first`,
     `app.boot`.
   - **Warm pass** — landing → click each session sidebar row × 2 laps.
     Triggers `nav.click` → `nav.session.ready` → `paint.first`.
   - **Goal pass** — creates a goal, clicks the dashboard button × 2 laps,
     then one `page.goto(#/goal/:id)` cold reload. Triggers `nav.click` →
     `nav.goal.ready`, `api.goal.fetch`, `api.goal.gates.fetch`,
     `api.goal.agents.fetch`.

### Outputs

- `tests/manual-integration/.perf-out/sidebar-nav-<ts>.json` — raw client
  entries + server `[timing]` lines tail.
- `tests/manual-integration/.perf-out/sidebar-nav-<ts>.html` — sortable
  per-span table for that run.
- `docs/perf/history/<short-sha>.json` — `{ commit, parentCommit, branch,
  timestamp, spans: { name: { p50, p95, p99, n, mean, max } } }`. One file
  per harness run keyed by HEAD short SHA. Committed to git so the
  cross-commit report renders at HEAD.
- `docs/perf/sidebar-nav-report.html` — regenerated at the end of every
  run by `scripts/perf-report.mjs`. Inline SVG line charts (p50 solid,
  p95 dashed) per canonical span across every committed history file,
  plus a first-vs-latest summary table.

Notes:

- `nav.goal.ready` has only `n=2` because the goal pass loops twice and the
  cold reload is captured separately as `paint.first` rather than under the
  click-driven nav span. This is enough to confirm instrumentation works;
  Phase 2 may want a richer goal pass.
- `reducer.rehydrate` is effectively zero because the seeded sessions have
  no transcript to rebuild. Real sessions with hundreds of messages will
  almost certainly be where the cost shows up; the harness's `messages: 0`
  detail makes that obvious in the trace.
- The harness re-seeds 10 fresh sessions before the warm pass because the
  server's archive-on-disconnect path sweeps the originals once the
  cold-pass WS connections close. This is a measurement artefact, not a
  product bug, and is the **pragmatic workaround** per the Phase 1
  finishing task's spec — we do not modify the archive code path.

## 3. Ranked hotspots (top 5 by p50 on the critical path)

Each comes with a one-sentence likely-cause hypothesis. These are seeds for
Phase 2; the goal is to keep or kill each based on a focused A/B.

1. **`nav.session.ready` p50 ≈ 89ms, p95 ≈ 193ms.** The dominant contributor
   on the click→ready critical path. Likely cause: synchronous `paint.first`
   (≈17ms) plus the WS-attach round-trip (≈36ms) plus
   `connectToSession` REST hydration (`api.session.fetch` ≈10ms) plus
   compositor settle. Subjective "click feels slow" complaints map here.
2. **`api.goal.gates.fetch` p95 ≈ 94ms, max 125ms.** The gates endpoint
   spikes on the very first call after goal creation; subsequent calls are
   <12ms. Likely cause: cold-cache file IO on the goal's `gates.json`, or
   workflow snapshot materialisation. Worth confirming with the server
   `io=` count from `[timing]` lines.
3. **`ws.attach` p50 ≈ 36ms.** Sits on the critical path of every cold
   session load. Likely cause: handshake + auth round-trip; the
   `outcome:auth_ok` detail confirms we reach auth. Phase 2 could test
   eager WS connect during the click handler (parallel with REST), or
   render-from-cached-snapshot-then-attach-WS.
4. **`paint.first` p95 ≈ 31ms, max 49ms.** The render frame itself. Likely
   cause: full `doRenderApp()` re-render on every state change including
   the entire sidebar + project tree. Phase 2 candidates: more granular
   re-render scoping, or deferring off-screen blocks via
   `requestIdleCallback`.
5. **`api.session.fetch` p95 ≈ 32ms.** Each session view hits
   `GET /api/sessions/:id` which ships the full transcript + tool inputs.
   For our seeded `messages: 0` sessions this is fast; real sessions with
   long transcripts will dominate. Phase 2 candidate per the goal spec:
   lazy-load tool content (the `/tool-content/:mi/:bi` endpoint already
   exists) so the initial payload is small.

The instrumentation will become more informative once it runs against a
realistic session corpus (long transcripts, multiple gates, an active
team). Phase 2 should re-baseline against such a corpus before committing
to optimisations.

## 4. Phase 2B A/B — lazy tool content (feature-flagged)

**Hypothesis.** `GET /api/sessions/:id` was thought to ship the full
transcript + tool inputs on every cold load; stripping large tool-call
content blocks above a low threshold (default 4KB) and lazy-loading them
on demand via the existing `/tool-content/:mi/:bi` endpoint should reduce
`api.session.fetch` payload size + parse time and, by extension,
`nav.session.ready`.

**What landed.** Behind the `lazyToolContent` perf flag (set via
`localStorage.bobbitPerfFlags=lazyToolContent`):

- Server: `GET /api/sessions/:id?stripToolContent=1` now opt-in includes a
  `messages` array with tool-call content blocks larger than the threshold
  replaced by the same `{ _truncated: true, _originalLength, preview }`
  shape that the WebSocket `truncate-large-content` path already emits.
  The existing renderer (`WriteRenderer` + `Messages.ts::_onLoadFullContent`)
  hydrates the full content on demand via `fetchToolContent()` — no new
  renderer code needed. Default response (no query param) is byte-identical
  to Phase 1.
- Client: the central `gatewayFetch` wrapper appends `?stripToolContent=1`
  to GET `/api/sessions/:id` URLs when the flag is on. Idempotent.
- Pure data-shape pinning test: `tests/session-strip-tool-content.test.ts`.

**Observation while implementing.** Reading the Phase 1 server route
revealed that `GET /api/sessions/:id` already returns **metadata only** —
the transcript is delivered over WebSocket via `get_messages`, not REST.
This means the stock `api.session.fetch` span measures a small
metadata-only payload (≈10ms p50, ≈32ms p95 with `messages: 0`). The
strip experiment now opts into shipping the transcript on REST so the
hypothesis can actually be tested; the comparison below records both.

**A/B measurement.** Run the manual harness twice on the same commit:

```bash
# Flag OFF (default) — metadata-only response
BOBBIT_PERF_FIXTURE_SIZE=large \
BOBBIT_PERF_HISTORY_TAG=flag-off \
  npx playwright test --config playwright-manual.config.ts \
  --grep "perf-sidebar-nav"

# Flag ON — transcript stripped + lazy-load placeholders
BOBBIT_PERF_FIXTURE_SIZE=large \
BOBBIT_PERF_HISTORY_TAG=flag-on \
BOBBIT_PERF_FLAGS=lazyToolContent \
  npx playwright test --config playwright-manual.config.ts \
  --grep "perf-sidebar-nav"
```

The harness writes `docs/perf/history/<sha>-flag-off.json` and
`<sha>-flag-on.json`; the cross-commit report shows the delta visually.

**Result.** _(Pending Phase 2A's richer transcript fixture — the
`messages: 0` seed produces a no-op strip. Numbers are populated once the
rich fixture lands and both passes are executed against it.)_

**Decision rule.** ≥100ms p50 reduction on `nav.session.ready` OR
`api.session.fetch` against the richer fixture → recommend flag default-ON
for Phase 3. Otherwise document as "tried, didn't pay off" and leave the
flag default-OFF but available.

## 6. Implementation notes (Phase 1 only — not for prose-pinning invariants)

- Client trace primitive: `src/app/perf-trace.ts`. Cost-when-disabled
  pinned by `tests/perf-trace.spec.ts`.
- Feature-flag helper for Phase 2: `src/app/perf-flags.ts`.
- Server timing extended in `src/server/server.ts:792` — log line format
  `[timing] METHOD path Xms bytes=B io=N`. Scoped to the five hot endpoints
  in the design doc.
- WS attach instrumentation lives in `src/app/remote-agent.ts::_connectWs`
  (open → `auth_ok`/`auth_failed`/`error`/`close`). The span closes once,
  whichever outcome arrives first.
- Sidebar click handlers in `src/app/render-helpers.ts` route through
  `openForNavItem` so both keyboard and mouse paths emit
  `nav.click` / `nav.session.ready` / `nav.goal.ready` from one place.
- `window.__bobbitRefreshSessions` and `window.__bobbitOpenForNavItem` are
  exposed from `main.ts` for the manual harness — landing-branch boot
  doesn't call `refreshSessions()`, so the harness kicks it explicitly.
- Manual harness: `tests/manual-integration/perf-sidebar-nav.spec.ts`.
  NOT in CI. Hard-fails (`process.exit(1)`) when any of the five canonical
  gate spans has zero samples.
- Cross-commit report: `scripts/perf-report.mjs` reads
  `docs/perf/history/*.json` and writes `docs/perf/sidebar-nav-report.html`.
  No deps, inline SVG, Bobbit CSS tokens.
