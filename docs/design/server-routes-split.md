# Splitting `src/server/server.ts` into per-domain route modules

**Status:** Design — implementation pending.
**Goal:** [`Split server.ts routes`](../../README.md#goals) (goal id `split-serv-35fa46fa`).
**Author:** coder-939a8e7a.

## 1. Problem

`src/server/server.ts` is **8,433 lines** and the top-churn file in the repo. The bulk is `handleApiRoute()` — a single ~6,873-line function (lines **1461 – 8334**) handling **209 REST routes** as nested `if/else` branches keyed off `req.method` and a hand-rolled regex applied to `url.pathname`.

Symptoms:

- Constant merge conflicts. Almost any feature touches it.
- Inconsistent error shapes. We grep up **452** `json({error…})` / `res.writeHead(4xx|5xx)` callsites versus only **54** uses of the existing `jsonError()` helper. Every handler was written ad-hoc.
- 28 helper / cross-project closures defined at the top of `handleApiRoute()` re-create themselves on every request.
- Helper functions wedged between route blocks (git, secrets, validators, static, manifest) have no obvious home.

This document specifies a mechanical split into per-domain route modules under `src/server/routes/` plus a thin dispatcher. **No behaviour changes**, modulo the (explicitly-allowed) normalisation of error shapes onto `jsonError()`.

## 3. Domain partition — rationale and edge cases

**Default rule:** group by first path segment after `/api/`. Deviations:

1. **Sessions split into 6 sub-modules.** A naïve "everything under `/api/sessions/*`" gives one ~3,000-line file. Split by responsibility: core CRUD (`sessions.ts`), per-session git (`sessions-git.ts`), bg processes (`sessions-bg.ts`), review (`sessions-review.ts`), content/transcript/skills (`sessions-content.ts`), proposals/drafts/titles (`sessions-proposals.ts`).
2. **Goals split into `goals.ts` + `goals-git.ts` + `gates.ts` + `tasks.ts` + `teams.ts`.** Originally the spec proposed one `goals.ts`. The team/swarm endpoints (`/api/goals/:id/(?:team|swarm)/...`) are ~250 lines and are tightly coupled to `TeamManager`. Keep them in `goals.ts` alongside CRUD/setup. The team-prompt + state-of-team endpoints need only `TeamManager`, fitting the same module without bloat. Cost endpoints span sessions/goals/tasks → dedicated `cost.ts`.
3. **`/api/internal/*` endpoints land in their natural domain rather than a single `internal.ts`** — `mcp-call` / `mcp-describe` → `mcp.ts`, `verification-result` / `user-question/submit` → `verifications.ts`, `test/replay-buffered-events` → `health.ts`. They are distinguished only by the trust model (locker bearer = sandbox token), enforced uniformly by the `requireInternalAuth()` helper, not by file layout.
4. **`/api/search/*` straddles two modules.** `GET /api/search` (per-project text search, hot path) lives next to sessions/goals data lookup → `routes/maintenance.ts` is wrong; move to `sessions.ts` or a dedicated `search.ts`. **Decision:** keep the hot search route in `sessions.ts` and move admin (`rebuild`, `stats`, `compact`) to `maintenance.ts`. Listed in the inventory under `sessions.ts` accordingly.
5. **`/api/pr-status-cache` (line 5239)** is a global GET, not goal-scoped. Routed via `goals-git.ts` because the cache is populated by the goal git endpoints; no domain mismatch.
6. **`/api/internal/test/replay-buffered-events/:id`** is dev-only test infra. Goes in `health.ts` (utility/diagnostic bucket).
7. **`routes/preview.ts`** owns `/api/preview/mount` (POST + GET) and the per-session SSE endpoint `/api/sessions/:id/preview-events`. The SSE endpoint physically lives under `/api/sessions/...` but its handler logic is 100% preview manifest plumbing — co-locate with the rest of the preview API.

### Ordering-sensitive matches

Plain Node `http`, no framework — order matters. Cases where prefix overlap would bite:

| Specific (must match first) | Generic |
|---|---|
| `/api/sessions/:id/wait` | `/api/sessions/:id` (DELETE, GET, PATCH) |
| `/api/sessions/:id/git-status` | `/api/sessions/:id` |
| `/api/sessions/:id/bg-processes/:bgid` | `/api/sessions/:id/bg-processes` |
| `/api/projects/:id/config/:variant` | `/api/projects/:id/config` |
| `/api/projects/:id/structured` | `/api/projects/:id` |
| `/api/goals/:id/tasks` | `/api/goals/:id` |
| `/api/goals/:id/gates/:gid/inspect` | `/api/goals/:id/gates/:gid` |
| `/api/goals/:id/team/agents` | `/api/goals/:id/team` |
| `/api/roles/assistant/prompts/:id` | `/api/roles/:id` |
| `/api/slash-skills/details` | `/api/slash-skills` |

The current code is correct because regex anchors are exact (`^...$`) — only one regex matches each pathname. We **preserve this property** by registering routes with full anchored regexes (or parameterised pattern strings the matcher converts to anchored regexes). The dispatcher then performs a linear walk in registration order, but because patterns are anchored only one match is possible per request; ordering is robustness-only, not correctness-relying.

Per-domain registration order is preserved within each module's exported `routes` array; module registration order in the dispatcher is preserved by the iteration order in `dispatcher.ts`. We document this contract in the dispatcher header comment and pin it with a unit test (see §9).

## 4. Dispatcher design

### 4.1 Types — `src/server/routes/types.ts`

```ts
import type http from "node:http";
import type { RouteDeps } from "./route-helpers.js";

/**
 * Per-request context passed to every route handler. All deps,
 * the parsed URL, the path-pattern match groups, and lazy body
 * accessor are pre-populated by the dispatcher.
 */
export interface RouteContext {
  req: http.IncomingMessage;
  res: http.ServerResponse;
  url: URL;
  pathname: string;
  /** Numeric/string capture groups. params[0] is the full match. */
  params: ReadonlyArray<string>;
  /** Sandbox token scope, if the caller authenticated as a sandbox token. */
  sandboxScope?: import("../auth/sandbox-tokens.js").SandboxScope;
  /** Lazy parsed JSON body. Returns null on malformed JSON (matches readBody). */
  readBody(): Promise<any>;
  /** Standard JSON response. */
  json(body: unknown, status?: number): void;
  /** Standard error response — every error path MUST go through here. */
  jsonError(status: number, err: unknown, extra?: Record<string, unknown>): void;
  /** All server singletons. */
  deps: RouteDeps;
}

export type RouteHandler = (ctx: RouteContext) => Promise<void> | void;

export interface Route {
  method: string;          // "GET" | "POST" | "PUT" | "DELETE" | "PATCH"
  /**
   * Either a literal string (anchored exact match) or a RegExp
   * (must be anchored with ^...$ — pinned by a unit test).
   */
  pattern: string | RegExp;
  handler: RouteHandler;
}
```

### 4.2 Matcher — `src/server/routes/match-route.ts`

```ts
export function matchRoute(
  method: string,
  pathname: string,
  routes: ReadonlyArray<Route>,
): { route: Route; params: string[] } | null {
  for (const r of routes) {
    if (r.method !== method && r.method !== "*") continue;
    if (typeof r.pattern === "string") {
      if (r.pattern === pathname) return { route: r, params: [pathname] };
    } else {
      const m = pathname.match(r.pattern);
      if (m) return { route: r, params: Array.from(m) };
    }
  }
  return null;
}
```

Linear scan is fine — 209 routes, anchored regexes, < 50 µs even at 1k req/s. We pay for clarity, not micro-perf.

### 4.3 Dispatcher — `src/server/routes/dispatcher.ts`

```ts
import { healthRoutes } from "./health.js";
import { sandboxRoutes } from "./sandbox.js";
import { projectsRoutes } from "./projects.js";
// … one import per domain module …

/**
 * Order: domain modules are registered in this exact order.
 * Within each module, routes are listed most-specific-first.
 * Both layers are necessary only for robustness — every pattern
 * is anchored, so at most one match exists per request.
 */
const allRoutes: ReadonlyArray<Route> = [
  ...healthRoutes,
  ...sandboxRoutes,
  ...projectsRoutes,
  ...sessionsRoutes,
  ...sessionsGitRoutes,
  ...sessionsBgRoutes,
  ...sessionsContentRoutes,
  ...sessionsProposalsRoutes,
  ...sessionsReviewRoutes,
  ...goalsRoutes,
  ...goalsGitRoutes,
  ...gatesRoutes,
  ...tasksRoutes,
  ...workflowsRoutes,
  ...rolesRoutes,
  ...toolsRoutes,
  ...preferencesConfigRoutes,
  ...modelsRoutes,
  ...imageGenerationRoutes,
  ...staffRoutes,
  ...skillsRoutes,
  ...previewRoutes,
  ...mcpRoutes,
  ...oauthRoutes,
  ...verificationsRoutes,
  ...costRoutes,
  ...maintenanceRoutes,
];

export async function dispatch(
  url: URL,
  req: http.IncomingMessage,
  res: http.ServerResponse,
  deps: RouteDeps,
  sandboxScope?: SandboxScope,
): Promise<void> {
  const m = matchRoute(req.method ?? "GET", url.pathname, allRoutes);
  const json = (body: unknown, status = 200) => {
    res.writeHead(status, { "Content-Type": "application/json" });
    res.end(JSON.stringify(body));
  };
  if (!m) { json({ error: "Not found" }, 404); return; }
  const ctx: RouteContext = {
    req, res, url, pathname: url.pathname,
    params: m.params,
    sandboxScope,
    readBody: () => readBody(req),
    json,
    jsonError: (status, err, extra) => {
      const e = err instanceof Error ? err : new Error(String(err));
      json({ error: e.message, stack: e.stack, ...extra }, status);
    },
    deps,
  };
  await m.route.handler(ctx);
}
```

`server.ts` then becomes:

```ts
await dispatch(url, req, res, routeDeps, sandboxScope);
```

— replacing the current 28-arg `handleApiRoute()` call site.

### 5.3 Auth — handled before dispatch

Auth lives in `requestHandler` (server.ts:670–760), already factored as a single guard producing the optional `sandboxScope`. No change. The dispatcher receives `sandboxScope` as an argument and stuffs it into `RouteContext`.

## 9. Risk analysis

| Risk | Mitigation |
|---|---|

## Appendix — scripts used

Inventory was generated with a Python script that:

1. Builds a map `varName → regex literal` by scanning `const xxxMatch = url.pathname.match(/.../)` and `const xxx = url.pathname === "..."` lines.
2. Walks every line in `[1461, 8334]` containing `req.method`, extracting `(method, path)` either from a `url.pathname === "..."` literal on the same line, a referenced `xxxMatch` variable, or — falling back — from the enclosing `if (xxxMatch) { ... }` block via brace-counted ranges.
3. Hand-corrects 9 routes that match by `startsWith / endsWith` chains against `/api/sessions/:id/(git-…|pr-…|commits|file-content)` since their patterns are not regex literals.

The script lives in this commit history at `build_inv.py` for re-runs.
