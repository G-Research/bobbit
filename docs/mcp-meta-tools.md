# MCP meta-tool aggregation

Bobbit collapses every MCP server into a single model-facing tool, instead of
exposing every individual operation. This page explains what changed, why, and
what stays the same.

For the full architecture and rationale, see
[docs/design/mcp-meta-tool-aggregation.md](design/mcp-meta-tool-aggregation.md).

## Why

The naive surface â€” one tool per `(server, operation)` pair â€” has three
concrete failure modes once a user has more than a handful of MCP servers:

1. **Context bloat.** Every operation contributes its full JSON Schema to every
   turn. 10 servers Ã— 30 operations Ã— ~400 tokens â‰ˆ 120 K tokens of tool
   definitions before the user has typed anything.
2. **OpenAI 128-tool cap.** `gpt-*` models reject the request entirely once
   the aggregated tool count exceeds 128. VS Code and Cursor have shipped
   public bug reports on this.
3. **One broken MCP server breaks the whole agent.** A single malformed tool
   schema or a `tools/list` timeout used to take down the agent for that
   turn â€” not just the misbehaving server.

The industry consensus (Anthropic's "Code execution with MCP", RAG-MCP,
Microsoft's mcp-gateway, Lunar.dev) converges on the same primitive: don't
expose individual operations to the model â€” expose **servers**, with
on-demand operation discovery. This page documents Bobbit's implementation of
that pattern at the gateway layer.

## What the model sees

For each registered MCP server, the model gets one meta-tool:

```
mcp_gr-halo(operation, args)
mcp_gr-jira(operation, args)
mcp_gr-github(operation, args)
```

Plus one shared discovery tool:

```
mcp_describe(server, operation?) â†’ JSON Schema
```

The meta-tool's input schema constrains `operation` with a typo-proof `enum`
of valid op names. `args` is a free-form object â€” per-operation validation is
the underlying MCP server's job.

The meta-tool's description (~80 tokens) lists operation names only; full
schemas live at `<stateDir>/mcp-tool-docs/<server>.md` and are returned by
`mcp_describe` on demand.

### Example agent invocation

```jsonc
// 1. Model doesn't know the args shape â€” discover first.
{ "name": "mcp_describe",
  "input": { "server": "gr-halo", "operation": "list-employees" } }

// â†’ returns { server, operation, inputSchema, description, summary }

// 2. Now invoke with the right shape.
{ "name": "mcp_gr-halo",
  "input": { "operation": "list-employees", "args": { "team": "platform" } } }
```

`mcp_describe` without an `operation` parameter lists every op on the server,
which is useful when the model is exploring.

## What stays the same

Internally, `mcp__<server>__<op>` remains the canonical routing identifier â€”
the meta-tool is a pure presentation-layer transformation. Nothing about
the wire protocol below the model changes:

| Layer | Behaviour |
|---|---|
| `<stateDir>/mcp-tool-docs/<server>.md` auto-docs | Unchanged â€” already per-server. |
| `/api/internal/mcp-call` dispatcher | Unchanged â€” body is still `{ tool: "mcp__<server>__<op>", args }`. |
| `tool-group-policies.yaml` keys (`mcp__playwright`, `mcp__nano-banana`, â€¦) | Unchanged â€” no migration. |
| `McpManager._toolNameMap` and `_parseToolName` | Unchanged. |
| `GET /api/mcp-servers` | Unchanged â€” still returns per-server status + per-op list for the UI. |
| `tests/grant-policy.test.ts` cascade fixtures (`mcp__pw__snap` etc.) | Unchanged â€” all 25 cases pass as-is. |

The agent-side meta extension (generated by `generateMcpMetaExtension` in
`src/server/agent/tool-activation.ts`) assembles the
`mcp__<server>__<op>` string from `(serverName, params.operation)` and POSTs
it to the existing `/api/internal/mcp-call` endpoint. The server has zero
knowledge of the meta-tool format.

## Failure isolation

A misbehaving MCP server no longer takes down the agent turn:

- **Per-call timeouts** â€” 10 s on `tools/list`, 30 s on `tools/call`,
  enforced in `src/server/mcp/mcp-manager.ts`. Wrapped at the manager level so
  the JSON-RPC client core (`mcp-client.ts`) stays untouched.
- **Schema validation at registration** â€” `isValidOperationSchema` in
  `src/server/mcp/mcp-meta.ts` filters malformed ops out of `tools/list`
  before they reach the meta-tool's `enum`. A bad op never poisons sibling
  ops on the same server, and never affects other servers.
- **Stub meta-tool when a server is down** â€” `writeMcpProxyExtensions` emits
  a stub `<server>.ts` extension whose `execute` returns a structured
  envelope:

  ```json
  {
    "error": "mcp_server_unavailable",
    "server": "gr-halo",
    "reason": "tools/list timed out after 10s"
  }
  ```

  The model gets a structured result and moves on. The agent turn never
  aborts at the protocol level.
- **Per-call error envelope** â€” the `/api/internal/mcp-call` catch block
  parses the inbound `tool` string and adds `server` / `operation` fields to
  the error response, so the meta extension surfaces a clean structured
  error.

## Two-level meta-tools (gateway servers)

Gateway-style MCP servers proxy several upstream sub-MCPs and emit
two-level operation names like `mcp__gr__ai-adoption__list-articles`. Bobbit
parses these structurally so each sub-namespace surfaces as its **own**
meta-tool, instead of collapsing into a single giant enum.

**Mental model:** MCP server = group, sub-namespace = tool, op = parameter.

The single source of truth is `parseMcpToolName()` in
`src/server/mcp/mcp-meta.ts`. Strip `mcp__<server>__`, split the remainder on
the **first** `__`:

| Input                                  | server       | sub           | op                |
|----------------------------------------|--------------|---------------|-------------------|
| `mcp__gr__ai-adoption__list-articles`  | `gr`         | `ai-adoption` | `list-articles`   |
| `mcp__gr__ai-adoption__create-article` | `gr`         | `ai-adoption` | `create-article`  |
| `mcp__gr__jira__get-queue`             | `gr`         | `jira`        | `get-queue`       |
| `mcp__playwright__click`               | `playwright` | (none)        | `click`           |
| `mcp__playwright__snap`                | `playwright` | (none)        | `snap`            |
| `mcp__foo__a__b__c`                    | `foo`        | `a`           | `b__c` (literal)  |

Meta-tool identity follows the same rule: `mcp_<server>__<sub>` when a
sub-namespace exists, `mcp_<server>` when flat. The dispatcher remains
`POST /api/internal/mcp-call` and still receives the original
`mcp__<server>__<sub>__<op>` (or `mcp__<server>__<op>`) string â€” the
meta-tool layer is purely the model-facing surface.

### Policy keys

Both keys may appear in `tool-group-policies.yaml` and per-role
`toolPolicies`. Most-specific wins (tool > group), and role overrides project.

| Scope                       | Key                       | Covers                                   |
|-----------------------------|---------------------------|------------------------------------------|
| Group (whole server)        | `mcp__gr`                 | every `mcp_gr__<sub>` meta-tool          |
| Tool (one sub-namespace)    | `mcp__gr__ai-adoption`    | `mcp_gr__ai-adoption` only               |
| Flat server                 | `mcp__playwright`         | `mcp_playwright` (= group = tool)        |

A legacy 3-segment key `mcp__server__op` is treated as a **tool key**
(sub-namespace) by default. Flat-server users with such keys are unaffected
because flat servers never carry sub-namespaces.

## Tools page UI

The Tools page surfaces one row per MCP **server** under a dedicated "MCP"
section, sibling to the existing builtin sections. Each server row mirrors a
built-in tool group:

- **Server header** (`data-testid="mcp-server-toggle"`): chevron, name, status
  pill (`connected` / `error` / `disconnected`), operation count, and a
  group-policy `<select>` (`data-testid="mcp-server-policy"`; key
  `mcp__<server>`). Click to toggle expansion.
- **Server expanded**: one **tool row** per sub-namespace
  (`data-testid="mcp-tool-row"`), each with its own `<select>`
  (`data-testid="mcp-tool-policy"`; key `mcp__<server>__<sub>` for sub,
  `mcp__<server>` for flat) and a click-to-expand toggle
  (`data-testid="mcp-tool-toggle"`). Flat servers render exactly one tool row
  whose name matches the server.
- **Operation rows** (deepest level, `data-testid="mcp-server-ops"`):
  read-only listing with description. Operation is a parameter, not a tool â€”
  no policy dropdown at this depth.

Both the server and tool selects route through `updateGroupPolicy(key, value)`
(no backend change â€” the endpoint already accepts arbitrary keys). This means
top-level tool count visible to the user matches the model's view, with the
option to drill into any server when needed.

## Policy resolution

Per-server YAML keys in `tool-group-policies.yaml` keep working untouched:

```yaml
# defaults/tool-group-policies.yaml
mcp__playwright: never
mcp__nano-banana: never
```

Both legacy per-op tool names (`mcp__playwright__snapshot`) and the new
meta-tool name (`mcp_playwright`) are mapped to the same prefix
(`mcp__playwright`) by `mcpPolicyPrefix()` in
`src/server/agent/tool-activation.ts`. The regex was extended to match BOTH
forms, so a single YAML key blocks the meta-tool **and** every per-op call.

Per-operation gating uses two enforcement layers:

- **Layer A (model-facing).** `computeToolPolicies` aggregates per-op
  policies into a single server-level policy on the meta-tool entry:
  `any 'ask' â†’ 'ask'`, `all 'allow' â†’ 'allow'`, otherwise `'never'`. This is
  what the tool-guard extension sees. `ask` fires once per server on first
  use; subsequent ops on the same server flow through.
- **Layer B (definitive).** `/api/internal/mcp-call` resolves the per-op
  policy on the inbound `mcp__<server>__<op>` string before dispatching to
  `mcpManager.callTool`, and returns 403 if it resolves to `never`. This
  guarantees `mcp__nano-banana: never` keeps blocking `nano-banana__generate_image`
  even if a future role granted the meta-tool wholesale.

Per-op `ask` policies aren't re-asked at Layer B â€” granted-once at the meta
level = granted-for-server. Users who need true per-operation gating should
use `never`, which is the only granularity that survives.

## Out of scope

Deliberate non-goals for this phase. Listed so future work doesn't accidentally
re-open the design discussion:

- **Code-mode / sandbox-mediated execution.** The model still emits
  `tool_use` blocks; we just collapse the names. A future phase can swap the
  dispatcher behind `/api/internal/mcp-call` to a TS-on-virtual-FS execution
  surface without changing what the model sees.
- **Per-op output truncation / summarisation.** Existing
  `truncateLargeToolContent` (>32 KB) handles big payloads in the chat
  renderer.
- **RAG retrieval over operations within a single server.** Helpful only
  above ~50 ops/server; revisit if a real user hits that.
- **Cross-server visual clustering** (e.g. all `gr-*` servers grouped). The
  Tools page MCP section is alphabetised by server name; an optional
  cluster-by-prefix UI is non-blocking future work.
